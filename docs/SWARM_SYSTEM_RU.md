# 🐝 Рой субагентов

Главный агент AAF - это Chief Agent, оркестратор. Он не должен выполнять рутинную работу (парсинг 50 страниц документации или чтение логов). Для этого он динамически спавнит субагентов. Субагенты используют более дешевые и быстрые модели (например, `gemini-3.1-flash-lite`).

## Классы субагентов

### 1. Workers (Одноразовые рабочие)
Выполняют конкретную задачу и "умирают", возвращая ресурсы в систему.
* `Researcher`: OSINT, глубокий поиск в сети, чтение статей.
* `SystemAnalyst`: Чтение локальных файлов, дебаг кода системы.
* `ChatSummarizer`: Чтение сотен сообщений из Telegram и составление выжимок.
* `Chronicler`: Архивариус. Анализирует граф и векторы, удаляя информационный мусор.

### 2. Daemons (Бессмертные фоновые процессы)
Живут вечно. Просыпаются по таймеру, проверяют условие и засыпают.
* `WebMonitor`: Мониторит сайты, цены или API. Если срабатывает триггер - экстренно будит главного агента через EventBus (через навык `trigger_swarm_alert`).

## 🕸️ Agentic Mesh (Эстафета)
Субагенты могут спавнить других субагентов. 
Например: `WebMonitor` нашел ошибку на сайте. Вместо того чтобы будить главного агента, он вызывает инструмент `delegate_task_to_swarm` и спавнит `SystemAnalyst`.
**Важно:** Субагенты не общаются друг с другом напрямую. Они передают данные через файлы в песочнице. Первый агент сохраняет данные через `write_local_file`, а в инструкциях второму агенту говорит: "Прочитай файл X через `read_sandbox_file`".
